## 2. 要解決的問題

Home Flow 第一版要解決 [docs/Plan.md](docs/Plan.md) 中的下列問題，並把問題轉成明確交付：

### 2.1 原始 CSV 不易讀

現況：家庭支出資料已存在 CSV，但表格是橫向日期結構，不利直接查看。

第一版解法：

- 提供 Web App 匯入流程
- 將 CSV 轉成結構化支出資料
- 以月報與分類分析替代原始表格閱讀

補充說明：

- 第一版的最小資料粒度來自 CSV 的單一儲存格
- 也就是一筆 日期 x 分類 x 金額 的紀錄
- 第一版不保證還原為真實世界的每一筆消費交易或發票級明細
- 若未來出現同一天同分類多筆消費，資料層允許同日同分類存在多筆紀錄，不先強制加總

### 2.2 無法快速回答核心財務問題

現況：使用者無法快速知道當月總支出、分類排行、波動與固定支出占比。

第一版解法：

- 提供當月總支出
- 提供分類排行與分類占比圖
- 提供與上月比較
- 提供固定支出標記與統計基礎

### 2.3 CSV 格式不完全固定

現況：未來仍會出現相近但不完全相同的 CSV。

第一版解法：

- 容忍欄位順序變化，但欄位語意需可辨識
- 容忍日期範圍長短不同
- 容忍分類列增減
- 支援分類別名映射

### 2.4 自動判定不一定可靠

現況：若只靠規則解析，錯誤分類會降低報表可信度。

第一版解法：

- 規則判斷為主
- 提供人工覆寫分類
- 提供人工覆寫固定支出標記

## 3. 產品定義

Home Flow 是一個家庭支出分析 Web App。

第一版定位如下：

- 主要用途是匯入家庭支出 CSV 並產生月度分析
- 使用者為 1 位管理者與 1 到 2 位只讀家人
- 不追求完整記帳系統
- 不處理銀行串接與多人即時協作

## 3.1 成本原則

第一版明確以免費資源優先為原則。

- 優先採用具免費額度或免費方案的服務
- 若同類方案皆可達成需求，優先選擇不需付費即可上線者
- 若某功能需要付費資源，必須先有免費替代方案或可延後到後續版本
- 第一版目標是在不增加固定月費的前提下完成上線

## 4. 使用者與角色

### 4.1 管理者

- 可登入系統
- 可上傳 CSV
- 可校正分類
- 可校正固定支出標記
- 可查看全部報表

### 4.2 家庭成員

- 可登入系統
- 以只讀方式查看報表
- 不可修改匯入資料與分類規則

## 5. 第一版功能範圍

### 5.1 In Scope

- Google Login
- 管理者與只讀角色
- 家人名單與角色由 repo 設定檔管理
- CSV 手動上傳
- 增量合併匯入
- 分類別名映射
- 固定支出規則判斷與人工覆寫
- 單筆支出明細查詢與過濾
- 月報摘要
- 分類排行
- 分類占比圖
- 與上月比較
- 月趨勢圖
- 匯入歷程列表
- RWD 介面，桌機與手機可用
- 免費額度內可部署與可運行

### 5.1.1 AI 功能邊界

- 若使用 LLM，第一版僅作為輔助功能，不應成為核心流程唯一依賴
- AI 功能第一版僅用於分類別名建議
- 第一版指定使用 Gemini 2.5 Flash-Lite 免費模型作為預設 AI 模型
- AI 功能失效時，CSV 匯入、校正、報表查詢仍需可正常運作
- AI 不可在未符合自動採納規則下直接寫入正式資料

補充說明：

- 每次匯入後自動對未匹配分類產生建議
- 若 Gemini 2.5 Flash-Lite 免費額度、穩定性或政策限制不符預期，AI 功能可整體停用或延後，不得改用付費模型作為第一版前提
- 高信心建議可自動寫入分類別名規則，預設門檻為 0.85
- 低信心建議需由管理者人工確認

### 5.2 Out of Scope

- 帳號註冊流程
- 角色與 allowlist 後台管理介面
- 銀行或信用卡串接
- 即時協作
- 推播通知
- 匯入版本回復
- AI 自動執行決策且不經人工確認

## 6. 核心使用流程

### 6.1 登入

1. 使用者使用 Google 帳號登入
2. 系統確認是否屬於允許名單
3. 若不在允許名單內，直接拒絕登入流程
4. 若在允許名單內，系統依角色導向管理頁或報表頁

### 6.2 匯入 CSV

1. 管理者選擇 CSV 檔案
2. 系統解析檔案與欄位語意
3. 系統將資料標準化為支出明細
4. 系統依日期與分類執行增量合併
5. 系統自動對未匹配分類執行 AI 分類別名建議
6. 系統標記解析異常、重複候選或需人工校正項目
7. 若同期間再次匯入且同日期同分類金額不同，視為更新候選，預設以新值為正確值
8. 管理者可在預覽階段改為保留舊值或略過
9. 管理者確認匯入預覽與衝突處理結果
10. 系統才正式寫入資料庫並記錄匯入歷程

### 6.3 校正資料

1. 管理者檢視未辨識或可疑資料
2. 管理者修正分類
3. 管理者調整固定支出標記
4. 系統即時反映到報表與明細

### 6.5 角色初始化

1. 系統透過 migration 或 seed 建立第一位管理者
2. 管理者名單與 viewer 名單以 repo 設定檔管理
3. 名單異動方式為修改設定檔後重新部署
4. 未列入 allowlist 的 Google 帳號不得登入系統
5. allowlist 設定檔僅適用於 private repo 情境

### 6.4 查看報表

1. 使用者進入月報首頁
2. 系統預設顯示最近月份摘要
3. 使用者切換月份或分類
4. 使用者查看分類排行、占比、上月比較、月趨勢與明細
5. 使用者可點擊圖表，聯動篩選明細列表與分類分析區塊

## 7. 成功標準

- 管理者可在 5 分鐘內完成登入、匯入、查看月報
- 家人可在不理解原始 CSV 的情況下讀懂支出結果
- 匯入後可立即回答當月總支出、主要分類、與上月比較
- 匯入格式只要仍屬相近格式，就不需修改程式即可處理
- 管理者可透過人工校正修正分類與固定支出標記
- 開發團隊可明確知道第一版明細是儲存格級紀錄，而非發票級交易

## 8. 技術與部署決策

### 8.1 已確認決策

- 前端框架：SvelteKit
- 產品型態：Web App
- 部署平台：Cloudflare Pages
- 交付流程：push 到 GitHub 後自動部署
- 資料庫：Cloudflare D1
- 登入方式：Google Login
- 成本策略：以免費方案與免費額度為優先
- 部署模型：靜態前端加 Serverless API，不採純靜態站

### 8.2 對應實作假設

- SvelteKit 使用 Cloudflare adapter
- 需要 Cloudflare Pages Functions 或對應 server runtime 支援登入與 API
- Google Login 採 OAuth 流程
- 允許名單與角色由 repo 設定檔提供，部署時載入
- allowlist 設定檔包含家人 email，因此 repo 必須維持 private
- 若使用 Google AI API key，必須只存在 server 端環境變數，不可下發到瀏覽器
- AI 呼叫需經由 server route 代理，不可由前端直接呼叫第三方模型 API
- 第一版若啟用 AI，預設目標模型為 Gemini 2.5 Flash-Lite 免費模型
- 第一版資料量與流量需控制在 Cloudflare 免費額度可承受範圍內
- 避免引入需要額外月費的第三方 SaaS
- 若 Gemini 2.5 Flash-Lite 免費額度不足，AI 功能需可關閉或降級，不影響主要功能

詳細設計見 [docs/prd/Architecture.md](docs/prd/Architecture.md)。
